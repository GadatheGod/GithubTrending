1. Extract frontend into deep modules

Strong

Files

js/app.js

168 lines, zero exports, six responsibilities mixed in one file

Problem

js/app.js is a monolithic module with no interface boundaries. It mixes configuration (LANGUAGE_COLORS), state management, data fetching, filtering logic, DOM rendering, and event binding. To understand any single behavior — say, how a search query filters repos — you must read the entire file.

applyFilters has six responsibilities: get repos, filter by search, filter by language, sort, mutate state, and call renderRepos. renderRepos references searchInput from closure, so it cannot be unit-tested without a DOM. The entire frontend exports zero functions — there is no seam to cross for testing.

Solution

Split js/app.js into three deep modules, each with a small interface and deep implementation:

  • js/filters.js — Filters module: interface filter( repos, { period, search, language, sortBy } ). Pure function. Zero DOM dependency.
  • js/renderer.js — Renderer module: interface render(container, repos, hasSearch). Accepts DOM element as parameter, not closure.
  • js/app.js — Orchestrator: wires modules together, binds events, manages state. Thin adapter layer.

Benefits

  • Leverage: filter() exercises all filtering/sorting logic through one method. One test covers search, language filter, and sort.
  • Locality: A bug in sort order lives entirely in filters.js. A bug in card rendering lives entirely in renderer.js.
  • Testability: filter() is a pure function — testable without JSDOM. render() accepts a container element — testable with a minimal JSDOM setup.
  • Depth: The Filters module has one method but complex logic behind it. The Renderer module has one method but complex HTML generation behind it.

2. Deepen the scraper — export and test scrapeTrending

Strong

Files

scripts/scrape-trending.js

The scrapeTrending(url) function is internal — not exported, not tested

Problem

scrapeTrending is the actual scraping logic: HTTP fetch + cheerio parsing + repo extraction. It is not exported, so tests must go through main() which writes to disk, or mock axios.get. The function does too much through its implicit interface.

The scraper's real seam is URL → repo array. Currently hidden inside scrapeTrending. By exporting it and making it accept a URL string and return a repo array, we create a testable interface. The HTTP dependency becomes an adapter concern.

Solution

  • Export scrapeTrending(url) as the scraper's public interface
  • Inject axios as a dependency (accept it as a parameter or use dependency injection)
  • Write tests that pass a mock axios and verify the parsed repo output
  • Keep main() as the thin adapter that wires axios to scrapeTrending and handles file I/O

Benefits

  • Depth: The scraper module now has one exported function with complex logic behind it (fetch + parse + extract)
  • Locality: Changes to the cheerio selectors live in one place. Changes to the HTTP layer live in one place.
  • Testability: Tests can mock axios and verify the scraping pipeline end-to-end without hitting GitHub
  • Deletion test: Deleting scrapeTrending would spread fetch+parse logic across main() — it earns its keep

3. Create a shared formatting module

Worth exploring

Files

scripts/scrape-trending.js (parseStars) · js/app.js (formatStars)

Inverse operations with no shared module — parseStars converts string to number, formatStars converts number to string

Problem

The scraper has parseStars("12.5k") -> 12500. The frontend has formatStars(12500) -> "12.5k". These are inverse operations operating on the same domain concept — star count formatting — but they live in separate files with separate implementations. If the formatting rules change (e.g., adding million display support), both must be updated independently.

Solution

  • Create shared/format.js with a Format module: formatStars(number) and parseStars(string)
  • Both the scraper (Node.js) and frontend can import from the same module
  • The Format module becomes a deep module with one interface surface and all star formatting logic behind it

Benefits

  • Leverage: One module, one test suite, covers both directions of star formatting
  • Locality: A new format rule (e.g., "show millions as 1.2M") lives in one file
  • Consistency: The scraper and frontend always use identical formatting rules
  • Depth: One interface (formatStars, parseStars) with all formatting logic concentrated inside

4. Deepen language color mapping

Worth exploring

Files

js/app.js (LANGUAGE_COLORS + getLanguageColor)

19-line constant + 2-line function, both shallow, both only used in rendering

Problem

LANGUAGE_COLORS is a 19-line lookup object. getLanguageColor is a 2-line function. Together they form a shallow module — interface complexity equals implementation complexity. They are embedded in app.js and only used by renderRepos.

Solution

  • Extract to shared/colors.js — a deep module with interface getLanguageColor(language)
  • Move LANGUAGE_COLORS inside the module's implementation
  • When app.js is split (Candidate 1), this becomes the Renderer module's internal color concern, not a global constant

Benefits

  • Locality: Adding a new language color lives in one file
  • Depth: The colors module has one method. If a fallback strategy or default color system is added later, the interface stays the same
  • Clean seam: The Renderer module imports getLanguageColor through a clean interface rather than inlining the color map

Top Recommendation

Tackle Candidate 1 first — extract the frontend into deep modules. This is the highest-leverage change because:

  • It creates the seams needed to test the largest untested area (js/app.js)
  • It naturally absorbs Candidates 3 and 4 — the formatStars and getLanguageColor functions become internal to the new modules
  • It produces immediate testability gains: the Filters module can be tested as a pure function without JSDOM
  • Candidate 2 (scraper deepening) can wait — the scraper already has decent test coverage through its exported functions

After Candidate 1 is done, the codebase will have clear seams between filtering, rendering, and orchestration. Each module will have a small interface and deep implementation. Tests will be natural to write because they cross clean interfaces rather than reaching into closure variables.
